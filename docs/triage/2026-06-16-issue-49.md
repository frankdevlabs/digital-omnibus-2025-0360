## Assessment

**T1-19 — transient, no action.** The Kamervragen RSS feed was added recently and
baselined successfully on 2026-06-15 (`tracker-state.yaml:96-99` carries that hash with
`last_seen_at: 2026-06-15`). The 2026-06-16 failure is the feed endpoint returning its own
HTTP 500 error page, not a routing/URL problem. The watchlist URL (`data/watchlist.yaml:156`)
is byte-for-byte the same query the cron prompt uses (`~/law-tracker/prompt.md:135`), so there
is nothing to correct. This is the first failure; the 14-consecutive-day investigate/pause
threshold is nowhere near. Verdict: leave the watchlist as-is and watch the next runs.

**T2-04 — orphaned tombstone, remove it.** T2-04 was already deleted from both
`data/watchlist.yaml` and the cron prompt's Tier-2 source table by the merged fix for
issue #16 (PR #30, 2026-06-03). Confirmed today: `grep T2-04 data/watchlist.yaml` → no
match; the cron prompt Tier-2 table (`prompt.md:150-157`) has no T2-04 row and no
"known-unreachable" list that would re-introduce it.

The **only** remaining reference is an orphaned comment line in the auto-maintained state
file: `data/tracker-state.yaml:109` —
`# T2-04: Cloudflare-blocked on every run; treat as unreachable, no hash stored.`

Issue #16's plan assumed this comment would "be overwritten on the next cron run." That
assumption was wrong: the cron only rewrites entries for sources it actually fetches, and
T2-04 is no longer fetched, so the tombstone is never touched. Each weekly source-health
run reads the prior state, sees the comment, and faithfully re-emits the "T2-04 … (known)"
row — generating permanent, signal-free noise. Removing the orphaned comment breaks that loop.

Note on the "Do not edit by hand" header on `tracker-state.yaml`: this edit removes dead
metadata for a deliberately-removed source, not a live hash entry. It does not falsify or
touch any source's hash/timestamp, and the cron regenerates the file from its own fetches
each run, so the edit is safe and within the spirit of that guard.
